# Suggested Learning Path

  • High-level overview of GitHub Actions (opens new window)
    • Key Concepts
      • Workflows (opens new window) - Automated process, such as build and deploy an application
      • Events (opens new window) - Casue workflows to execute. Event Types (opens new window) include:
        • Scheduled like CRON
        • Manually create by clicking a button
        • Triggered by an action ex. opening a pull request
      • Jobs (opens new window) - Set of steps comprising a logic unit of work in your process.
        • Can have more than one in a workflow
        • Can run serially or in parellel
        • An example might be a job that runs tests
      • Actions (opens new window) - Custom application that provides functionality, can write your own or use pre-built ones from the Marketplace (opens new window)
      • Step -
      • Runners (opens new window) - Server virtua/hosted/on-prem where the workflow is executed
      • Secrets (opens new window) - Sensitive environment variables like passwords, API keys, and tokens. GitHub provides a mechanism for securely secrets with your source code. Secrets can be stored at the organization, repository and environment level. Once created secrets can then be used in your workflows.
      • Environments (opens new window) - Are collections of settings (allowed branches, secrets, and protection rules.) An environment in GitHub likely correlates to the environment your workflow will act against.
        • By using a named environment secret you can have a single workflow file, that uses different secrets based on the environment the workflow is executed in.
  • Quickstart (opens new window) - Create your first workflow
  • Write your own action (opens new window) - Chances are you will not often need to do this, but understanding how the actions you are using a built will give you a better understanding of how to use them and why something might not work as expected.
  • Continuous Integration (opens new window) - This will guide you in creating your first workflow.
  • GitHub Script (opens new window) - Create a workflow that uses the GitHub script action (more about this below).


# Comparisons with Jenkins

  • Both allow creating custom/reusable functionality. Jenkins often accomplishes this via plug-ins in GitHub they are Marketplace Actions, Composite & Reuasable Workflows
  • Both support describing a process in a document that can be checked into source control. In Jenkins they are called pipelines (Jenkinsfile), GitHub Actions calls them workflows (.gitub/workflows/{your-workflow}.yml)
  • Both support timed and manually triggered jobs. Jenkins can be triggered by checking code into GitHub (and can likely do more via Plug-ins); however, GitHub Actions natively supports a more diverse and granular set of events, and seems better suited for automations
  • Both Jenkins and GitHub Actions support "secrets". GitHub's approach seems more secure.


# Tips, Resources, & Useful Actions

  • GitHub Checkout Action (opens new window) - Checks your code out of your repo so it can be used in your workflow. Likely you will use this in most workflows.
  • Terraform Setup (opens new window) - Sets up the terraform CLI making it easy to execute your IAC
  • GitHub Cache Action (opens new window) - Allows you to cache artifacts across runs in the same branch. Useful to speed up your builds.
  • GitHub Script Action (opens new window) - Exposes GitHub's RESTful APIs inside a Script block of a workflow "step". This is very useful for automations
    • RESTful API documentation (opens new window) - General API documentation
    • Octokit Documentation (opens new window) - NodeJS implementation of GitHub RESTful API.
  • GitHub Action Org (opens new window) - Owner of all the GitHub developed repositories containing Actions related code
  • GitHub Actions Documentation (opens new window)
  • Starter/Template Workflows (opens new window) - Contains templated workflows for common tasks
  • If you add a Repository Secret with the name ACTIONS_STEP_DEBUG and the value of 'true' this will turn on debugging logs for your Workflows.
  • YAML is all about indentation, and it is easy to cause yourself headaches due to missing a space. Use a YAML linter like this (opens new window) for VSCode
